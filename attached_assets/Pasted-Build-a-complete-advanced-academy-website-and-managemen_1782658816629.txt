────────────────────────────────────────

- type text check (type in 
  ('complaint','inquiry','announcement','reply')) 
  default 'complaint'

Parent can only send type: complaint or inquiry.
Admin can send type: announcement or reply.
Teacher receives type: announcement only.
i Filter messages in dashboard by type.

────────────────────────────────────────
FIX 5 — RENAME public_results TABLE
────────────────────────────────────────
Rename public_results table to result_highlights 
everywhere in schema, policies, queries, and UI.
This avoids confusion with the private results table 
that stores student exam marks.

result_highlights = public poster/image based achievements 
shown on the public Results page.

results = private per-student exam marks, 
only visible to admin, teacher, parent, and student.

────────────────────────────────────────
FIX 6 — ADD is_published TO HOMEWORK TABLE
────────────────────────────────────────
Teachers should be able to save homework as draft 
before publishing to students.

Add this column to homework table:

are already defined. Just use the Supabase client 
to interact with them.

RULE 2 — DO NOT USE A SEPARATE SERVER
────────────────────────────────────────
Do NOT create:
- A separate Express server
- A separate Node.js server
- A separate API server
- A separate backend folder with its own entry point
- Any standalone server file like server.js or index.js

All server logic must live inside Next.js only:
- Use Next.js Server Actions for form submissions 
  and mutations
- Use Next.js Route Handlers (app/api/) for 
  any API endpoints needed
- Use Next.js middleware for auth protection

There is only ONE project. One Next.js app.
No separate backend. No separate server process.

────────────────────────────────────────
RULE 3 — ALL DATA MUST BE IN SUPABASE
────────────────────────────────────────
Do NOT use:
- Local JSON files as database
- SQLite
- MongoDB
- Firebase
- Any other database

All data must be stored in Supabase:
- Students, teachers, parents, classes in Supabase tables
- Images uploaded via Supabase Storage
- Auth handled by Supabase Auth
- All queries via Supabase client

────────────────────────────────────────
The admin dashboard is the most important part.

Admin panel must actually work:
- Add student → saves to Supabase students table
- Add teacher → saves to Supabase teachers table
- Add parent → saves to Supabase parents table
- Mark attendance → saves to Supabase attendance table
- Add fee record → saves to Supabase fee_records table
- Upload gallery image → saves to Supabase Storage
- Create notice → saves to Supabase notices table
- View admission inquiries → reads from Supabase
- Create exam and results → saves to Supabase

Do not build fake/hardcoded admin panels.
Every button and form must connect to real Supabase data.

────────────────────────────────────────
The public website pages must also be fully working:

- Home page loads real stats from Supabase settings table

- Results page loads real public result highlights

- Admissions page form saves inquiry to Supabase
- Contact page form works and saves to Supabase

────────────────────────────────────────
RULE 6 — ENVIRONMENT VARIABLES
────────────────────────────────────────
Use these environment variables.
Never hardcode keys in code.

NEXT_PUBLIC_SUPABASE_URL=your_supabase_url
NEXT_PUBLIC_SUPABASE_ANON_KEY=your_supabase_anon_key
SUPABASE_SERVICE_ROLE_KEY=your_service_role_key
NEXT_PUBLIC_SITE_URL=your_site_url

SUPABASE_SERVICE_ROLE_KEY must only be used 
on the server side (Server Actions, Route Handlers).
Never expose service role key in client components.

────────────────────────────────────────
RULE 7 — AUTHENTICATION
────────────────────────────────────────
Use Supabase Auth only.
No custom JWT. No NextAuth. No Passport.js.

After login, redirect by role:
- admin or owner → /dashboard/admin
- teacher → /dashboard/teacher
- parent → /dashboard/parent
- student → /dashboard/student

Protect all dashboard routes with Next.js middleware.
Public pages are accessible without login.
